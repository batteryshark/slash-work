# Incubate Work Ideas

Preserve a possibility without turning it into a commitment. Minimize memory,
initiation, and review costs while retaining the user's original intent.

## Keep the boundary clear

- Create one Idea per distinct possibility. Do not create a meta-project or one
  large "project proposals" Idea as a container.
- Prefer a workspace-owned Idea when no real project owns the possibility. Do
  not create a folder solely to hold the Idea.
- Treat detail as context, not commitment. A long design note can remain an Idea.
- Create a project or executable work only after the user explicitly adopts the
  Idea and authorizes that separate action.
- Use a Note only for passive reference material. Use an Idea when the material
  describes something the user might choose to pursue.

## Guide without interrogating

Draft from the material already provided. Ask at most one or two short questions
only when the missing answer would materially change the record. Never require
priority, urgency, complexity, estimates, due dates, a complete plan, or a
definition of done merely to preserve the Idea.

Gently establish these points when the source supports them:

1. What is the possibility?
2. Why might it matter to the user?
3. Why is it not being pursued now, if known?
4. What real condition would make it worth revisiting?
5. What is the smallest evaluation step that could reduce uncertainty?

Do not fabricate missing answers. Leave optional content empty or label an
unknown plainly instead of assigning homework.

## Shape the Idea

Preserve specific details, links, constraints, prior research, and the user's
wording. Condense repetition without flattening the personality or motivation
behind the Idea.

Use Slash Work's canonical Idea sections:

- **Opportunity**: the possibility in plain language.
- **Why It Might Matter**: personal value, usefulness, or curiosity.
- **Hypothesis**: the lightest current belief, not a promise.
- **Unknowns**: only consequential unanswered questions.
- **Potential Shape**: approaches or boundaries already worth retaining.
- **Evidence**: existing artifacts, links, experiments, or prior work.
- **Risks and Constraints**: meaningful limits without catastrophizing.
- **Next Evaluation**: one small research or validation step, not a backlog.
- **Outcome**: leave empty until an outcome exists.

Convert premature task lists into context or a single next evaluation. Preserve
useful technical detail, but do not ask the user to maintain a speculative
execution plan.

## Choose state calmly

- **Open**: preserve the possibility when no decision is needed.
- **Exploring**: use only while evaluation is actually happening.
- **Not now** (`deferred`): use when the user has consciously parked it and a
  truthful reason is known. Record that reason without shame.
- **Proposed**: use when it is ready for an explicit commitment decision.
- **Adopted**: use only after the user chooses to pursue it.
- **Closed** (`declined`): use when the user decides it is no longer worth
  pursuing; preserve the reason and record.

Default to **Open** when intent is unclear. Do not infer **Adopted** from detail,
enthusiasm, age, or prior effort. Do not call a quiet or deferred Idea stale.

Set a revisit date only when the user gives a meaningful calendar date or a
genuine time-based trigger. Otherwise leave it unset. Describe non-calendar
triggers in the body, such as another project finishing, hardware becoming
available, or a dependency maturing.

## Work with Slash Work

When reading or mutating a `.work` workspace, also follow the sibling
`slash-work` skill. Use `work agent instructions ideas.create` or the relevant
Idea operation as the installed Work version's authority. Preserve workspace
scope, project ownership, lifecycle history, and agent-intent semantics.

Idea evaluation authorizes analysis only. It never authorizes implementation.
After creating or updating an Idea, confirm concisely what was preserved, its
scope and state, and what was deliberately not created.
